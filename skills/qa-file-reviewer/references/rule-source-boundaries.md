# File Reviewer - 规则来源与不过拟合边界

本文档定义 `qa-file-reviewer` 的规则来源、落点和禁止事项，用于防止把单个样本文档里的问题硬编码成源码规则。

## 来源优先级

1. 公司明确标准：来自 `company-review-standards.md` 或经确认的公司文件要求。
2. 可配置术语库 / 规则库：来自受控 JSON、术语表或部门维护的配置文件。
3. 文档结构通用模式：可跨文档稳定复现的结构性问题。
4. LLM 语义判断：需要阅读全文、跨段理解或语言知识的问题。
5. 人工复核：证据不足、锚点不稳、配置未覆盖或需要业务背景的问题。

## 能力分配

| 问题类型 | 默认承担方 | 允许自动批注条件 | 不应进入源码的内容 |
| --- | --- | --- | --- |
| 页边距、字号、页脚、表格重复表头 | `format` | 公司标准明确且能稳定定位 | 单个文件的视觉偏好 |
| 项目号残留 | `project_number` | 当前项目号和合理引用语境可判定 | 产品名录、项目名录、历史特例 |
| 断裂引用、组别编号、第三语言字符集残留、同格双语方法缩写遗漏 | `content_consistency` | 规则是结构模式，不依赖具体样本文本 | 固定句子、固定试验条件、固定产品组合 |
| 中文、英文、双语、语义一致性 | 调用 agent 语义复核 | 原文锚点唯一、证据明确、置信度不低 | 固定错词、固定错译、固定英文句式枚举 |
| 需要业务上下文或锚点不稳的问题 | 内部诊断 / 跳过计数 / 日志 | 默认不写入 Word；只有显式全文级 `global_summary` 可追加到文末 | 任何伪造高置信定位、写成确定错误或增加人工复核负担的 fallback |

## 禁止硬编码样本 literal

不得把单个样本文档中的以下内容直接写成源码判断：

- 文件名、项目号、产品名、部门名、人员名。
- 某一句中文或英文原文。
- 某个拼写错误、错译短语、试验条件组合。
- 某个表格编号、章节编号或组别编号的固定值。
- 某个审批页、模板页中的完整拼接字符串。

例外仅限：

- literal 是公司标准原文，例如固定页脚文本。
- literal 来自可配置术语库 / 规则库，并且规则读取配置而不是写死在源码中。
- literal 只作为测试 fixture、负例或文档说明存在，不参与生产规则判断。

## 泛化准入条件

新增确定性规则前必须同时满足：

- 规则能表达为结构模式、格式标准、字段约束或配置项。
- 至少一个测试使用合成变体，不能只复用最初触发问题的原文。
- 规则输出包含可追溯 `rule_id`、`evidence` 和稳定锚点。
- 对不满足稳定锚点、证据不足、低置信或建议空泛的问题，默认过滤出 Word；保留内部跳过原因。只有明确全文级 `global_summary` 且建议可执行的问题可追加到文末全局审核意见。

## 可配置术语库使用边界

可配置术语库适合存放：

- 公司批准术语及译名。
- 禁用译法及替代建议。
- 文件编号、部门代码、受控模板名称等稳定枚举。
- 可审计的严重度默认值和规则说明。

可配置术语库不应存放：

- 单个样本文档才出现的错句。
- 未经确认的业务推断。
- 不能解释来源的临时字符串。
- 需要 LLM 或人工判断上下文的事实性结论。

## Fixture 沉淀方式

样本文档只用于沉淀样本类型和覆盖场景：

- 制度正文类。
- 格式式样 / 模板类。
- 表单 / 记录类。
- 外部公文或审计管理类。
- 研发报告类。

fixture 记录文件名和样本类型即可；除非用户明确授权，不复制真实 Word 内容，不把真实正文片段写入源码规则。
