# 公司 Word 审核标准

本文档只保留公司审核的硬规则。凡是这里没有明确写出的内容，一律不要臆测；默认过滤出 Word，仅在内部诊断中保留跳过原因。只有明确全文级 `global_summary` 或已被质量门禁允许的真实核对项，才可作为用户可见批注。

## 1. 格式标准

### 1.1 字体与字号
- 中文字体：宋体。
- 英文字体：Arial。
- 正文字号：小四，12pt。
- 表格内容字号：五号，10.5pt。
- 审批页标题字号：四号，14pt，居中。
- 页脚字号：8号。

### 1.2 段落与对齐
- 正文：单倍行距，段前段后 0.75 行，左对齐。
- 表格：单倍行距，左对齐，垂直居中。
- 参考资料、记录、附录：单倍行距，段前段后 0.75 行，左侧缩进两字符。

### 1.3 页面设置
- 纸张：A4。
- 页边距：上 2.5cm、下 2.5cm、左 3cm、右 2.5cm。
- 装订线：左 0.5cm。
- 页眉：1.5cm。
- 页脚：1.75cm。
- 页脚固定文本：`公司机密-仅供内部使用 Company Confidential-Internal Use Only`。

### 1.4 标题层级
- 标题层级必须连续，不跳级。
- 一级到五级标题的样式必须与公司格式式样一致。
- 需要中英文并列时，中文在前，英文在后。
- 中英文分行时使用软回车，不要拆成新段落。

### 1.5 页眉页脚和审批页
- 签批页标题、职位、签名、日期等字段格式必须符合公司样式。
- 正文页眉应保留文件类型、文件编号、版本号等信息。
- 所有文件都应检查公司 Logo 和页脚是否齐全。

### 1.6 表格与列表
- 表格标题、表头、跨页重复表头、表格宽度和表内字体都要符合公司格式。
- 5 级标题之后如需继续分层，优先使用项目符号。

## 2. 语言标准

### 2.1 中文
- 检查错别字、多字、漏字、语法错误、表达歧义和标点误用。
- 专业术语必须准确、统一。
- 中文语境使用中文标点，不混用英文标点。

### 2.2 英文
- 检查拼写、语法、时态、大小写和缩写使用。
- 专业术语必须准确、统一。
- 首次出现的缩写应尽量给出全称。

## 3. 双语一致性标准

- 中文在前、英文在后。
- 中英文摘要、标题、图注、标签、说明必须语义一致。
- 关键数据、单位、百分比、结论必须对应一致。
- 同一术语的翻译必须统一。

## 4. 内容一致性标准

- 正文与表格、图形、图注中的数据必须一致。
- 引用的表号、图号、章节号必须正确。
- 摘要、结论必须有正文数据支撑。
- 正文、结果、结论之间不得出现明显矛盾。

## 5. 项目号标准

- 文档标题附近、页眉、元数据中的项目号应优先视为当前项目号。
- 正文中的项目号只有在明确引用、比较、历史说明、来源说明时才允许出现非当前项目号。
- 无上下文说明的跨项目号、模板残留和异常重复都应标记为问题。

## 6. 严重等级

- `关键`：影响数据准确性、合规性或结论有效性的问题。
- `主要`：明显的格式错误、重要内容不一致、明显语言问题。
- `次要`：单个错别字、轻微格式偏差、局部不统一。

## 7. 审核意见 / 批注内容

### 7.1 Word 用户可见批注

Word 批注面向业务审核人，应简洁、可定位、可执行。默认只显示：

- `发现`：合并用户可读的问题判断和依据，不重复展开内部字段。
- `原文`：可在 Word 中定位的原文片段。
- `建议`：直接可执行的修改动作。

用户可见批注不得暴露内部段落号、表格坐标、JSON 文件名、SDK 错误、堆栈信息、内部枚举或模型执行状态。除明确全文级 `global_summary` 外，不能稳定插入到原文位置的问题默认过滤出 Word。

### 7.2 内部结构化记录

`review_result.json`、`comment_plan.json` 和诊断日志可以保留更多结构化字段，用于追溯和调试：

- 问题类型 / category
- 位置 / anchor_locator
- 严重等级 / severity
- 原文证据 / original、evidence_quote、evidence
- 修改建议 / suggestion
- 来源和过滤原因 / source、review_basis、skip_reason

## 8. 人工复核条件

- 图像、扫描件、低质量 OCR 结果无法确认时。
- 规则未明确覆盖的特殊文档结构时。
- 需要结合业务背景才能判断是否错误时。
- 需要方案、记录、LIMS、样品台账等外部资料才能确认时，默认保留为内部诊断或跳过原因，不写成 Word 确定错误。
