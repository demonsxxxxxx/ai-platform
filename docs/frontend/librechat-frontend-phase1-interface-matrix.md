# LibreChat Frontend Phase 1 Interface Matrix

Status: Phase 1 local implementation evidence companion. This document maps the current frontend route surface to existing ai-platform route families and records which standalone pages remain fail-closed until backend contracts exist.

## Rule

Phase 1 may use only current ai-platform public/admin projections or compatibility routes that are already backed by the ai-platform control plane. Compatibility routes are explicit migration debt, not product authority. A page that still depends on legacy product-authority APIs is kept as a Phase 2 unavailable surface and must not call its API client from active `TabContent`.

## Phase 1 Guardrails

- `frontend/web/src/components/layout/AppContent/TabContent.tsx` must not import backend-missing management panels.
- `frontend/web/src/components/layout/AppContent/phase1SurfacePolicy.ts` is the shared authority for route permissions and navigation visibility.
- `frontend/web/src/auth/aiPlatformPermissions.ts` is the shared authority for frontend effective permissions derived from backend principal strings.
- Active frontend login must use `/api/ai/auth/login`, `/api/ai/auth/me`, and `/api/ai/auth/logout`. `/api/auth/*` token login remains compatibility-only.
- `/auth/register`, `/auth/callback`, `/auth/reset-request`, `/auth/reset-password`, `/auth/verify-email`, and `/auth/pending` must stay fail-closed until ai-platform owns those backend contracts.
- Active profile info must stay read-only against the ai-platform principal; username edit, password change, avatar upload/delete, and OAuth token fragment handling remain Phase 2.
- Browser requests use the ai-platform httpOnly cookie. Active frontend code must not attach stale `Authorization: Bearer` tokens to same-origin API calls.
- Active sidebar session loading must use the ordinary `/api/sessions` list without Phase 2 `project_id` or `favorites_only` filters until backend filter semantics are authoritative.
- `admin:status` grants Admin Runtime, tool-policy inventory, and active-notification visibility only; it does not grant settings CRUD, user/role admin, MCP server lifecycle, or model admin writes.
- `agent:use` supports the chat loop and chat-bound selectors. It does not expose standalone management pages.
- `/skills`, `/mcp`, `/agents`, `/models`, and `/notifications` use `frontend/web/src/services/api/phase1Projection.ts`; they must not import legacy management panels or legacy product-authority API clients.
- The global app shell must not import `useSettings` or call `/api/settings/*`; legacy settings CRUD remains dormant Phase 2 work.
- `/shared/:shareId` must stay fail-closed in Phase 1 and must not import `frontend/web/src/components/share/SharedPage.tsx` or call `/api/share/public/*` until ai-platform owns the share projection.
