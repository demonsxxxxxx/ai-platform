## Design And Boundaries

The MCP directory remains the ordinary-user surface. It renders only redacted directory data and no management control, form, endpoint, header, command, environment-key, or credential value. The administrator path is gated exclusively by `canManageMcpLifecycle` and uses `useMCP` lifecycle methods against the current backend routes. Enable and disable intentionally stay on shared `PATCH /api/mcp/{name}/toggle`, which the controller re-verified as tenant-scoped and non-expanding.

Administrators get an icon-only add action, an explicit empty-state action, per-server edit and enable/disable actions, and a confirmed delete action. Creation and editing use the existing MCP form in a contained modal. The form sends department IDs, allowed roles, and role quotas through the current lifecycle DTO. Since responses are write-only projections for connection material, an edit does not repopulate endpoint or credential fields; those values must be entered again to change configuration.

Historical stale evidence: earlier harness failures were caused by the synthetic bootstrap aborting before installing fetch mocks, which redirected the browser to `/auth/login`; generation-6 and generation-7 browser runs were superseded by later harness contracts. Current browser evidence is the strengthened generation-8 six-case synthetic run under the local harness only.

No backend, auth, schema, dependency, CI, Docker, 211, deployment, unrelated frontend surface, or locale outside English and Chinese is in scope.
